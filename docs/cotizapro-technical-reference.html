Módulo 01
Autenticación y Multi-tenancy
Sistema de identidad basado en JWT con aislamiento de datos por organización vía Row-Level Security (RLS) a nivel de base de datos.
Arquitectura de Identidad
JWT sin estado — tokens revalidados en cada request vía middleware
RLS en todas las tablas — el DBMS rechaza queries cross-tenant
Sesiones gestionadas por Supabase Auth (GoTrue)
Refresh token automático en middleware de Next.js
Server Components usan createServerClient() — cero estado cliente
Client Components usan createBrowserClient() aislado
Sistema de Roles
Rol Crear/Editar registros Eliminar Admin del catálogo Gestionar equipo Campos personalizados
owner ✓ ✓ ✓ ✓ ✓
admin ✓ ✓ ✓ ✓ ✓
member ✓ ✗ ✗ ✗ ✗
viewer ✗ ✗ ✗ ✗ ✗
Endpoints
POST /api/auth/logout Invalida sesión y limpia cookies
POST /api/team/invitations Invitar miembro por email (expira en 7 días)
POST /api/team/invitations/accept Aceptar invitación y unirse a la organización
GET /api/team/members Listar miembros del equipo con roles
PATCH /api/team/members/[id] Cambiar rol de un miembro
DELETE /api/team/members/[id] Remover miembro del equipo

Módulo 02
Gestión de Clientes
CRUD completo de clientes con búsqueda full-text, tags, empresa y protección ante eliminación si existen cotizaciones activas.
Campos del Modelo
clients { id UUID -- PK generado automáticamente organization_id UUID -- FK → organizations (multi-tenant) name TEXT -- Nombre completo del contacto company_name TEXT? -- Empresa / razón social email TEXT? -- Con validación de formato phone TEXT? -- Teléfono / WhatsApp address TEXT? -- Dirección de instalación tags TEXT[] -- Etiquetas clasificatorias notes TEXT? -- Notas internas created_at TIMESTAMPTZ -- Auto updated_at TIMESTAMPTZ -- Trigger automático }
Endpoints
GET /api/clients Listado con búsqueda (name/email/phone), paginación limit/offset

PATCH /api/clients/[id] Actualización parcial con validación

Módulo 04
Motor de Cotizaciones
Core del sistema. Cotizaciones multi-item con cálculo automático de subtotal, descuento, IVA 16% y total. Numeración automática. Ciclo de vida completo desde borrador hasta cobro.
Ciclo de Vida — Estados
draft → sent → viewed → accepted → en_instalacion → completado → cobrado
Terminales alternativos: rejected expired
Lógica de Cálculo Automático
subtotal = SUM(item.quantity × item.unit_price) descuento = subtotal × (discount_rate / 100) iva = (subtotal - descuento) × 0.16 -- IVA 16% fijo total = subtotal - descuento + iva -- Numeración automática vía función PostgreSQL: quote_number = COT-{YEAR}-{NNN} -- ej: COT-2026-001 -- Secuencia reiniciada por año, única por organización
Endpoints
GET /api/quotes Listado con filtros: status, date range, client_id — paginado
POST /api/quotes Crear cotización con items en transacción atómica. Rollback si fallan items.
GET /api/quotes/[id] Detalle completo: quote + items + client + notifications + payments
PATCH /api/quotes/[id] Actualizar — recalcula totales, reemplaza items completo
DELETE /api/quotes/[id] Solo permitido en estado draft
POST /api/quotes/[id]/send Generar PDF → subir a Storage → enviar Email + WhatsApp simultáneamente

Módulo 05
Registro de Pagos
Módulo de cobranza parcial o total por cotización. Soporta múltiples abonos con validación de saldo, tipos de pago y generación de recibos en PDF.
Capacidades Clave
Múltiples abonos por cotización (anticipo + saldo)
Validación: monto no puede exceder saldo pendiente
Tipos: anticipo, parcial, liquidación, devolución
Métodos: efectivo, transferencia, tarjeta, cheque, otro
Campo received_by para auditoría de quién cobró
Cálculo automático de total_paid y total_pending
Generación de recibo PDF descargable por pago
Solo owner/admin pueden eliminar pagos registrados
Endpoints
GET /api/quotes/[id]/payments Lista pagos + total_paid + total_pending
POST /api/quotes/[id]/payments Registrar pago — valida que no supere saldo, status 201
DELETE /api/quotes/[id]/payments/[paymentId] Eliminar pago — solo owner/admin, status 204
GET /api/quotes/[id]/payments/receipt/[paymentId] Descargar recibo PDF del pago específico

Integraciones Externas
Pipeline completo para entregar cotizaciones: generación de PDF profesional → almacenamiento en cloud → envío multicanal (Email + WhatsApp) con tracking de entrega.
Generación de PDF — jsPDF
Header con número de cotización y logo de empresa
Tabla de items con cantidad, precio unitario y subtotal

Datos del cliente (nombre, teléfono, email)
Fecha de emisión y validez en formato es-MX
Notas y términos con saltos de línea automáticos
Retorna Buffer — listo para upload y attachment
Sin dependencias de navegador — 100% servidor
WhatsApp — Twilio
Envío via API de Twilio WhatsApp Business
Mensaje con link público al PDF en Supabase Storage
Formato automático de números (+52 México)
Soporte para media (imagen de portada del PDF)
Consulta de status de entrega (delivered/read/failed)
Override de número destino (whatsapp_override)
Email — Resend
Template HTML profesional con estilos inline
PDF adjunto directo al email
Botón CTA para ver cotización online
Campos del cliente en el cuerpo del email
Override de destinatario (email_override)
Tracking de notificación en quote_notifications
Storage — Supabase
Bucket documents público para PDFs generados
Path: org_id/quote_id/cotizacion.pdf
URL permanente y pública para links en WhatsApp
RLS: solo usuarios autenticados pueden subir

El endpoint retorna { success, results: { email, whatsapp }, pdf_url }. Cada canal falla independientemente — si el email falla, el WhatsApp continúa y viceversa.

Módulo 07
Sistema de Recordatorios de Seguimiento
Recordatorios programados para seguimiento post-servicio. Soporta recurrencia automática — al completar un recordatorio, el sistema crea automáticamente el siguiente.
Tipos de Recordatorio
maintenance — mantenimiento preventivo
follow_up — seguimiento de cotización
renewal — renovación de contrato
custom — recordatorio personalizado
Prioridades / Estados
Prioridad: baja / normal / alta / urgente
Estado: pendiente → enviado → completado
Pospuesto: snooze N días configurable
Cancelado: soft-delete con registro
Recurrencia Automática
Caso de uso real: al instalar un minisplit, crear recordatorio de mantenimiento anual. Al marcarlo como completado, el sistema crea automáticamente el siguiente con fecha +12 meses. Sin intervención manual.
-- Configuración de recurrencia: is_recurring BOOLEAN recurrence_interval_months INT -- ej: 6, 12 -- Función SQL disparada al completar: create_next_reminder_occurrence(reminder_id) -- Clona el recordatorio con nueva fecha calculada
Endpoints
GET /api/reminders Listado con filtros: status, type, priority, due_only, days_ahead
POST /api/reminders Crear — vincula a client_id, quote_id (opcional), category
GET /api/reminders/due Recordatorios vencidos en próximos N días (?days_ahead=7)
POST /api/reminders/[id]/complete Marcar completado — activa creación del siguiente si is_recurring
POST /api/reminders/[id]/snooze Posponer N días — body: { days: number }
PATCH /api/reminders/[id] Editar título, fecha, prioridad, notas
DELETE /api/reminders/[id] Cancelar recordatorio

Módulo 12
Webhooks para Integraciones Externas
Sistema de webhooks salientes para notificar sistemas externos (CRMs, ERPs, apps internas) sobre eventos del ciclo de vida de cotizaciones, clientes y recordatorios.
Eventos Disponibles
quote.created — nueva cotización creada
quote.sent — cotización enviada al cliente
quote.accepted — cliente aceptó
quote.rejected — cliente rechazó
client.created — nuevo cliente registrado
client.updated — datos de cliente actualizados
reminder.due — recordatorio vencido
reminder.completed — recordatorio marcado completado
Características Técnicas
Secret key por suscripción (crypto.randomBytes) para verificación HMAC
Configuración de reintentos: max_retries y retry_delay_seconds
Log de entregas en webhook_delivery_logs (status, latencia, respuesta)
Estados de entrega: pending / success / failed / retrying
Solo admin/owner puede gestionar suscripciones
Función SQL get_pending_webhook_deliveries() para procesamiento
Endpoints
GET /api/webhooks Listar suscripciones activas de la organización
POST /api/webhooks Crear suscripción — body: { url, events[], description }

Módulo 14
Cron Jobs — Automatización
Procesamiento automático diario de recordatorios vencidos. Envía notificaciones multicanal sin intervención manual. Configurado para Vercel Cron o servicios externos.
Flujo de Ejecución Diaria
-- 9:00 AM UTC / 3:00 AM hora México GET /api/cron/reminders-check Authorization: Bearer {CRON_SECRET} 1. Obtener recordatorios vencidos (hoy + 1 día) via get_due_reminders() 2. Por cada recordatorio: a. Enviar Email via Resend al cliente b. Enviar WhatsApp via Twilio al cliente c. Actualizar status → 'sent' d. Registrar notification_sent_at (evita duplicados) 3. Retornar: { processed, sent, failed, errors[] }
Seguridad y Compatibilidad
Header Authorization: Bearer CRON_SECRET requerido
Compatible con Vercel Cron (plan Pro/Team)
Compatible con GitHub Actions (workflow diario)
Compatible con cron-job.org (plan gratuito)
Idempotente — notification_sent_at previene reenvíos
Multi-organización — procesa todos los tenants en una ejecución
Endpoints
GET /api/cron/reminders-check Ejecución automática — requiere CRON_SECRET en Authorization header
GET /api/health Health check — retorna { status: 'ok', timestamp }

Módulo 15
Seguridad y Rate Limiting
Capas de seguridad aplicadas a nivel de middleware, API route y base de datos. Diseño defense-in-depth con múltiples puntos de validación.
Capas de Seguridad
Middleware — refresca JWT, verifica sesión en rutas protegidas
CORS configurado — whitelist de orígenes + soporte localhost dinámico
RLS en DB — aislamiento de tenant imposible de bypass desde la API
Zod en cada endpoint — validación estricta de inputs antes de cualquier query
Role enforcement — verificación de rol en operaciones admin
Error logging — logger.security() para intentos no autorizados
No secrets en cliente — SUPABASE_SERVICE_ROLE_KEY nunca expuesto
Parameterized queries — Supabase SDK previene SQL injection
Rate Limiting
-- lib/rate-limit.ts defaultApiLimiter: 100 req / 15 min por IP -- Activo solo en producción (NODE_ENV === 'production') -- Desactivado en desarrollo/tests para no bloquear E2E -- Respuesta al superar límite: HTTP 429 Too Many Requests Retry-After: {seconds}
Logging Estructurado
logger.api(method, path, status, count) — todas las respuestas
logger.security(msg, context) — intentos no autorizados
logger.database(op, table, context) — queries críticas
logger.error(msg, error, context) — errores con stack trace
Stack de Tecnologías de Seguridad
Supabase Auth (GoTrue) PostgreSQL RLS JWT RS256 Zod Schema Validation CORS Middleware Rate Limiting (Production) TypeScript Strict Mode Server Actions Only
